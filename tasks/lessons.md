# Lessons — tangki-duty

## 互動深度（2026-06-18 回饋）
- **症狀**：儀式/問答「點幾下就過、亂點也沒差」，缺遊戲性與回饋。
- **規則**：互動要有「狀態 + 後果 + 物理回饋」。長按要有量表動畫、搖晃要全畫面回應、選擇要牽動可累積的儀表並回頭影響後續。不要做「點 N 下 → 完成」的空互動。
- **參考**：使用者給的「姓名戰鬥陀螺」= 目標是那種 rAF 物理感（慣性、透視、撞擊粒子、震屏、重量感），套到起乩體驗，不是真的做陀螺。

## 效能
- **症狀**：操作卡頓延遲。
- **根因**：每次 selectGod 都整段 renderGodGrid() 重建 DOM + 每次 scrollIntoView smooth；possession 灑 72 顆 DOM particle；document.on* 全域 handler。
- **規則**：選神改 transform 位移的 carousel（不重建 DOM）；事件用 addEventListener + passive；canvas 特效控制粒子數；避免每次互動重建大量節點。

## 文字可讀性
- 字太融背景 → 一律加深色 scrim / text-shadow，字級加大；字體統一 Iansui（Google Fonts）。

## 信眾資料
- 同一輪不可重複人物；同一人物要能問不同問題（問題與人物解耦）。
- 圖片邊角醜 → 邊緣羽化（mask radial-gradient feather）。

## 圖片不被切（關鍵）
- 信眾/神明圖都是 864×1152 = 精確 3:4。要「完整不切又無黑邊」：**讓容器 aspect-ratio 等於圖片比例（3/4）再用 object-fit:cover** → 滿版、零裁切、零 letterbox。
- 錯誤做法：容器比例 ≠ 圖片 → cover 會切、contain 會留黑邊。object-position:top 只是少切一點，治標。
- 橫式例外（wangye 1248×832）：用 contain + 填色背景。

## 字體（使用者指定 Iansui）
- 「改字體」= 全站可見字都要換，包含**標題/CTA/強調**，不只 body。我第一次只把 body 設 Iansui，標題還留 Ma Shan Zheng/Noto Serif → 被打槍。
- 三個字體變數 --hand/--serif/--brush 全指向 Iansui。canvas fillText 需 webfont 已載入 → 啟動時 document.fonts.load('16px Iansui')。
- Iansui 只有 400 字重，font-weight:900 會 faux-bold，注意別太糊。

## 信眾資料（第三輪）
- **名字隨機 bug**：GIVEN 是男女混池、不分性別亂抽 → 男生叫女生名。一定要依 gender 分名池。
- 使用者要「一人一圖固定角色」：每個 character 寫死 名/性別/人設/圖/故事，名字與圖 100% 對應。缺圖在資料寫 `genPrompt`（真實攝影風格提示詞），使用者拿去生圖、依檔名補回。
- 反差人設：精緻韓系/日系雜誌網美 站在傳統香火街景；8+9 典型穿搭（超緊身上衣、瘦皮猴、緊身褲、金鍊、刺青半甲、阿志頭）。

## 互動深度（每階段都要真互動）
- 「降駕前準備」不能只點一下 → 每步驟一個 canvas 2D 仿 3D 小遊戲（同 ritual 那套引擎）。
- 問答要多階段對話＋角色記憶，選擇影響後續話與結局，不能選啥都沒差。

## 每神明回答要有個性（第三輪）
- 現在 genReplyOptions 用通用 PERSONA_LINES，五位神明講一樣 → 要依 god 給專屬口吻：三太子火爆、關聖威嚴文言、媽祖慈柔、濟公嬉笑、王爺公文體。persona(敢衝/穩健/斂財/唬爛) × god 都要有對應台詞。

## 神諭文字可讀性
- 選完出現的字背景太花 → 加深色實底 scrim、字體加大、停留時間拉長（讀得完）。

## 介面遊戲化（第四輪後）
- 使用者明確：**不要格柵系統/列表，很醜**。主選單要做成「地圖」樣子，每個功能是地圖上的地點(pin/建築)，要能回地圖。所有頁面盡量場景化、遊戲化，少用方塊清單。
- 地圖背景圖也要寫進待生圖文件。

## 數值要對得上、要清晰（第六輪）
- 扣錢機制要一致：in-shift 事件扣的是「本班香火（未入袋）」，要講清楚、結算逐項對得上（毛收／事件增減／開銷／利息／淨利／名聲）。
- 事件要說清楚「誰、為什麼」——哪位信眾、因你之前哪個回答而來。

## 技能與深度（第六輪）
- 技能要有可點的技能樹（解鎖/升級），且效果能「改變玩法」，不只是加數值。
- 升級效果要感受得到（信眾+2 真的多人、收入差異明顯）。
- 神諭偶爾要「真的針對信眾問題」回應（不一定好，但讓玩家知道神偶爾真懂），而非永遠固定三種。
- 臨時事件效果要顯著、機會命運感。
- 加錢特效要全螢幕金幣落下、金光閃閃，更浮誇。

## 經濟平衡必須用實際數字驗算（第七輪）
- 致命 bug：steady(好好答/honest) 的 incense 點數=0 → 玩得好賺$0，逼人 scam，但 scam 掉 rep ＝死亡螺旋、根本轉不了。
- 開局開銷 1364 >> 任何可能收入(~500) → 必虧。重設：開局開銷要 < 認真玩的收入，難但可能（正期望值但薄利）。
- 每次調平衡都要 `node` 跑 economy.js 算「開局最佳收入 vs 開銷」確認可正循環，別憑感覺。
- 特效門檻要對齊「實際金額級距」（金幣雨 amount>=3000 但開局只賺 60-180 → 永遠不觸發）。門檻要相對化。
- 要有「現有現金 ×幾%」的高風險高報酬管道（賭一把/投資/法會），給爆發成長的出口。

## 第八輪需求重點
- 移除路人信眾，改成「更多固定角色」撐起無限性；隨名聲提升解鎖更猛圈子（偶像/議員/立委/市長/堂主/財團老闆/coser…），各有精心故事線，要能一路撐到買最後豪宅。
- 待生圖文檔：已生圖的移除、新角色加入。
- 可幫助信眾 = price/30000（與商品故事無關，是「這筆錢可幫 N 人」的反諷換算）。使用者問了→這就是答案，目前合理可保留。
- 賭博獨立成地標「奇乩娛樂城」：完整背景、多個賭博設施、情緒價值，非三按鈕；圖入待生圖。
- 事件節奏：選完出現的字常太快、看不完 → 改「點一下才下一步」；事件氣泡停久一點。
- 信義區把妹 → 「乩der」(仿 Tinder)：左右滑、右滑要滿足對方暗示的奢侈品需求(線索藏在自介/商品說明，不講白)才配對；右滑進好友清單，幾秒後可能「接受」(推播)解鎖聊天，未接受則卡著；聊天用選項(不能自己打字)，每人對話線設計好——認真的也會狂要奢侈品，反諷社會。

## 諷刺基調・雙重觀感（最高指導原則，反覆被提醒）
- 文案**絕不明講詐騙/坑/騙**。要「司馬昭之心，句句不提句句不離」——表面是正經的招財/開運/產品介紹，壞只藏在弦外之音。
- 目標：**8+9 看了覺得帥**（甚至想拿來當起乩練習模擬器）；**聰明人看了懂在反諷／反串／地獄梗／黑色幽默／政治不正確**。
- 反例（太露骨、被打槍）：「她兒子的命停在你家樓下」「這支錶是消災金買的」直接點破。
- 正解：用「開運話術／成功學／命理術語」包裝，讓讀者自己會心一笑，而非作者明說。
- 不要用套版格式（不要寫「背後故事：」標籤、不要每個商品同結構），要像真的商品文案自然敘述、長短不一。

## 驗證・CDP headless 技術細節
- Node 內建 `WebSocket`(undici) 的 message handler 收到的是 `MessageEvent`，`ev.data` 可能是 Blob/物件，不是字串。要 `typeof ev.data==='string' ? ev.data : await ev.data.text()` 再 `JSON.parse`，否則 `JSON.parse('[object MessageEvent]')` 噴錯。（這次踩了兩次）
- 驗證帶門檻的功能（如 乩der 配對）要**跑兩種庫存**：一份「全頂級」證明配對成功＋好友＋推播接受＋聊天，一份「窮酸」證明被拒＋線索提示＋禮物選項 disabled＋牌堆滑完後重洗。單一庫存只證明一半。
- 種存檔：navigate → `localStorage.setItem` → `Page.reload`，App 才會讀到 seed。
- hub.js 內 `vibrate()` 過去只在 main.js 定義卻在 hub 用 → runtime ReferenceError（build 不會抓）。已在 hub.js 補本地 `vibrate` helper。教訓：跨模組用的 helper 要 import 或各自定義，別靠「剛好是全域」。

## UX：少分層
- 不要為了一個功能多包一層選單（例：廟務經營別藏在「乩天宮→子選單」）。能一鍵到的就一鍵。

## 諷刺基調
- 表面全帥（8+9 看了想當練習模擬器），反諷藏在結算數據/成就/免責聲明。聰明人看得出黑色幽默、地獄梗。
- 問答是精髓：隱藏正解傾向 + 三人格神諭 + 三儀表連動（香火💰/滿意度😊/元神🔥）+ 結算人格鑑定。
